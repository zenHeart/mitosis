# Phase E: 目标演进

## 演进时机

- 当前目标完成（所有 stage PASS）— 由 Phase D 触发
- 用户新增需求 — 执行中追加
- 技术方案需要重大调整 — 调整现有 criteria
- 遇到阻塞，需要改变策略 — 降级或拆分

## 演进评估

在给出演进建议之前，先评估以下维度：

### E1: 完成度评估

```
当前目标完成度 = pass_count / total_count

- 100% → 目标完成，建议归档 + 新建下一目标
- 70-99% → 接近完成，建议继续当前目标
- 40-69% → 部分完成，建议评估剩余 work 是否值得继续
- <40% → 大量未完成，建议分析阻塞原因
```

### E2: 阻塞分析

检查 `consecutive_failures` 和 fail 的 criteria：

```
连续失败 >3 次的 criteria：
  - 如果是技术阻塞（外部依赖未就绪）→ 建议降级或拆分
  - 如果是需求不明确 → 建议澄清后重试
  - 如果是范围过大 → 建议拆分为子目标
```

### E3: 演进模式推荐

| 场景 | 推荐模式 | 操作 |
|------|---------|------|
| 100% 完成 | 归档新建 | Phase D 归档 → 询问下一目标 |
| 新增需求 | 增量追加 | Phase X Add → 追加 criteria → 继续 |
| 需求变更 | 增量调整 | Phase X Adjust → 更新 criteria → 重置 state |
| 连续失败 + 范围过大 | 降级拆分 | Phase X Descope → 移入 backlog |
| 用户明确说换个方向 | 替换目标 | Phase X Replace → 归档 → 新建 |

## 演进原则

- 演进是正常的，不是失败
- 保持 goal.md 的清晰和准确
- 每次演进都记录为什么变化（通过 Phase X 的变更记录）
- 不在 scope 外添加功能（除非 Phase X 明确分类为 Add/Replace）
- **已完成的 criteria 不会被删除，只会被归档**

> **注意：** Phase X（CRUD 路由）负责判断变更类型和执行增量操作。Phase E 只负责"是否建议演进"和"推荐哪种模式"。具体的 CRUD 操作在 Phase X 完成。
