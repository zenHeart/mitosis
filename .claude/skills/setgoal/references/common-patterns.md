# 常见模式

## 模式 1: 新增功能（标准模式）

```
Stage 1: 类型定义 + API 扩展
  ─────────────────────────────────────────────────────
  - 验收项 1: 新增类型定义
  - 验收项 2: 新增 API 函数
  ─────────────────────────────────────────────────────
                          │
                          ▼  PASS
Stage 2: 存储层实现
  ─────────────────────────────────────────────────────
  - 验收项 1: Pinia store 定义
  - 验收项 2: 持久化逻辑
  ─────────────────────────────────────────────────────
                          │
                          ▼  PASS
Stage 3: UI 组件开发
  ─────────────────────────────────────────────────────
  - 验收项 1: 新组件
  - 验收项 2: 集成到现有页面
  ─────────────────────────────────────────────────────
                          │
                          ▼  PASS
Stage 4: 端到端验证
  ─────────────────────────────────────────────────────
  - 验收项 1: Playwright 测试
  - 验收项 2: 构建验证
  ─────────────────────────────────────────────────────
```

## 模式 2: Bug 修复

```
直接定位到对应 stage
- 评估 → 理解问题
- 执行 → 最小化修复
- 验证 → 确认修复
- 归档 → 记录到 docs/retrospectives/
```

## 模式 3: 技术债务清理

```
Stage 1: 评估债务范围
  - 列出所有问题
  - 优先级排序
Stage 2: 逐项修复
  - 一次一个
Stage 3: 验证无回归
```

## 模式 4: 探索性任务

```
评估 → 如果可行
  - 设计阶段
  - 实现阶段
  - 验证阶段
如果不可行
  - 归档发现
  - 演进目标
```

## 模式 5: 部分实现恢复

```
适用场景: goal.md 存在，部分代码已写好，state 丢失或不准确

流程:
  Phase 0: Pre-flight → 检测到上次未归档，执行归档
  Phase A3: 代码核查 → 发现哪些 criteria 已实现
  Phase A4: 文档核查 → 比对 docs/decisions.md（ADR）和代码
  Phase A5: 状态同步 → 更新 .goal-state.json（已实现的标记为 pass）
  Phase X: CLASSIFY → 检查用户是否有新增/调整
  → 进入 Phase B 执行剩余 criteria

关键: 此模式下 Phase A 的代码核查（A3-A5）是核心，
      不是简单的"读 state → 找 pending"，而是"读代码 → 校正 state"
```

## 模式 6: 增量演进

```
适用场景: 目标执行中需要新增或调整需求

决策流程:
  Phase X 判断变更类型:
    新增需求 → Add → 追加 criteria 到对应 stage → 继续执行
    调整需求 → Adjust → 更新 criteria 描述 → 重置受影响 state → 继续
    降级需求 → Descope → 移入 docs/goals/backlog.md → 继续
    替换目标 → Replace → 归档旧目标 → 新建

关键: 所有变更都记录到 goal.md 的"变更记录"区域，
      state 的 goal_history 数组记录每次变更。
```
