## 1. 匿名用户 Gallery 首页路径（Task #1）

### 1.1 测试路径
```
访问 https://mitosis.zenheart.site
  → 显示 Gallery 首页
  → 展示已部署的应用卡片
  → 每个卡片显示：应用名称、描述、"打开"按钮
```

### 1.2 发现的问题

#### P2 - 缺少匿名用户引导 CTA
**现象**：Gallery 页面只展示应用卡片，但没有明显的"登录"或"创建应用"引导按钮。
**影响**：新用户不知道如何创建自己的应用，需要自己找到 GitHub OAuth 入口。
**建议**：
- 在页面顶部或底部添加"使用 GitHub 登录创建应用"的引导区域
- 或者将"+ 新建对话"按钮在匿名用户登录前可见

#### P3 - Gallery 应用名称显示
**现象**：应用卡片的中文名称显示正常，但长名称可能在小屏幕上被截断。
**影响**：可读性降低。
**建议**：添加 `text-overflow: ellipsis` 的同时，在移动端改为单列布局。

### 1.3 正常的功能
- ✅ 应用卡片布局清晰，网格排列合理
- ✅ "打开"按钮明确可点击
- ✅ 底部"🧬 在 Mitosis 中继续迭代"链接正确
- ✅ Footer 信息完整

## 2. 登录流程与 Workspace 聊天页面（Task #2）

### 2.1 测试路径
```
点击电源按钮（⏻）
  → 显示登录页面
  → 点击"使用 GitHub 登录"
  → OAuth callback（404.html 拦截）
  → sessionStorage 存储 code
  → authStore.init() 交换 token
  → 进入 Workspace
```

### 2.2 发现的问题

#### P1 - 双重错误消息显示
**现象**：当构建失败时，UI 先显示原始错误 `"❌ 创建任务失败: Failed to fetch"`，然后才显示友好的错误消息 `"认证失败：Token 无效或已过期，请前往 Setup 页面更新。"`
**影响**：用户看到技术性错误（Failed to fetch），然后又被中文消息覆盖，造成混淆。
**根本原因**：前端可能在多个地方显示错误（例如：`formatUserError()` 和原始错误 fallback）。
**建议**：
- 统一错误显示逻辑，只显示一层用户友好的消息
- 技术性错误记录到日志，不直接显示给用户

#### P1 - 进度指示器在失败后仍显示
**现象**：构建失败后，进度条仍显示"分析需求..."，造成"任务仍在处理"的错觉。
**影响**：用户不知道任务是否真的失败了，或者是否还在处理中。
**根本原因**：进度条状态在错误发生时没有正确重置或显示失败状态。
**建议**：
- 在错误发生时，将进度条标记为"失败"状态
- 添加红色高亮或"失败"标签
- 提供"重试"按钮

#### P2 - 重复的澄清消息
**现象**：在聊天历史中，相同的澄清消息（"我需要确认一下你的意图..."）出现两次。
**影响**：聊天历史冗余，用户体验差。
**根本原因**：可能是 triage 系统被触发两次，或者消息被重复添加到历史。
**建议**：
- 检查 triageMessage() 的调用时机，避免重复触发
- 在添加澄清消息前检查最近一条消息是否相同

#### P3 - 发送按钮缺少文字标签
**现象**：发送按钮只有一个图标（纸飞机），没有文字标签"发送"。
**影响**：新用户可能不清楚按钮的用途。
**建议**：添加 `aria-label="发送消息"`，或在按钮旁边添加"发送"文字。

### 2.3 正常的功能
- ✅ OAuth flow 正确通过 404.html 处理
- ✅ Auth state 管理正常（sessionStorage）
- ✅ Workspace 布局清晰（侧边栏 + 消息列表 + 输入框）
- ✅ 输入框自动聚焦，支持多行输入
- ✅ "添加图片"按钮存在
- ✅ Triage 系统正确识别构建意图（"创建一个简单的计算器应用" → 创建应用）
- ✅ 错误消息脱敏（不暴露原始 API 响应）

## 4. 移动端响应式体验（Task #4）

### 4.1 测试环境
- **设备**: iPhone SE（模拟）
- **分辨率**: 375x667
- **DPR**: 1x（CSS 像素）

### 4.2 发现的问题

#### P0 - 缺少侧边栏切换按钮（CRITICAL）
**现象**：在移动端（375px），没有汉堡菜单按钮来切换侧边栏的显示/隐藏。
**影响**：
- 侧边栏固定在 44px 宽，始终显示，占用主内容区域空间
- 用户无法查看完整的会话列表（应用名称被截断）
- 用户无法隐藏侧边栏以获得更大的聊天区域
**根本原因**：`Workspace.vue` 和 `Gallery.vue` 中没有实现移动端侧边栏切换逻辑。
**建议**：
- 添加汉堡菜单按钮（固定在左上角）
- 侧边栏在移动端改为抽屉模式（从左侧滑入）
- 添加遮罩层（overlay），点击遮罩关闭侧边栏
- 默认情况下侧边栏隐藏，点击汉堡菜单显示

#### P1 - 侧边栏宽度固定
**现象**：侧边栏在移动端固定为 44px 宽，只显示图标，不显示文字。
**影响**：用户无法看到完整的应用名称，只能看到图标。
**建议**：当侧边栏展开时，宽度应增加到 250-280px，显示完整的会话名称。

#### P2 - 聊天消息气泡宽度
**现象**：在移动端，消息气泡的 max-width 可能过大，导致文字换行过多。
**影响**：可读性降低。
**建议**：将消息气泡 max-width 改为 90%（相对于视口宽度）。

#### P3 - 输入框高度
**现象**：在移动端，输入框的高度可能不足以容纳多行文字。
**影响**：长消息输入困难。
**建议**：输入框最小高度应为 44px（触摸友好），并且支持自动增高。

### 4.3 正常的功能
- ✅ 输入框正常接收文字
- ✅ 发送按钮在输入内容后正确启用
- ✅ 发送按钮触摸区域足够（44px）
- ✅ 消息列表正常显示
- ✅ 页面没有横向滚动

**文件**: `src/components/Workspace.vue`
**影响**: 用户看到技术性错误，然后又被中文消息覆盖

#### 4. 重复的澄清消息
**文件**: `src/composables/useStepFun.ts` 或 `src/stores/session.ts`
**影响**：聊天历史冗余
**修复方案**:
- 在添加澄清消息前，检查最近一条消息是否相同
- 使用 `lodash/debounce` 或类似机制防止重复触发

#### 5. 消息气泡 max-width 在移动端过大
**文件**: `src/components/Workspace.vue`
**影响**：可读性降低
**修复方案**:
```css
@media (max-width: 768px) {
  .message-bubble {
    max-width: 90vw; /* 使用视口宽度单位 */
  }
}
```

### P3 - LOW（锦上添花，不影响核心体验）

#### 6. 匿名用户缺少引导 CTA
**文件**: `src/components/Gallery.vue`

- 或者在应用卡片下方添加"创建你的第一个应用"引导

#### 7. 发送按钮缺少文字标签
**文件**: `src/components/Workspace.vue`
**影响**：新用户可能不清楚按钮用途
**修复方案**:
```vue
<button :disabled="!canSend" aria-label="发送消息">
  <svg>...</svg>
  <span class="sr-only">发送</span>
</button>
```
